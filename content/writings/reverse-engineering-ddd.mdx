Most teams discover their domain model by reading the code, not the documentation. This isn't a failure — it's an opportunity.

## The Problem With Forward-Design

Traditional Domain-Driven Design assumes you start with a deep understanding of the domain. You gather experts, draw bounded contexts on whiteboards, and define ubiquitous language before writing a single line of code.

In reality, most enterprise systems evolve organically. The code accumulates domain knowledge that never makes it back to any architectural document. The codebase becomes the de facto source of truth.

```typescript
// This aggregate reveals business rules
// that no one documented
interface OrderAggregate {
  id: string;
  lines: OrderLine[];
  status: OrderStatus;
  appliedPromotions: Promotion[];

  // Why is this here? Because orders can only
  // be modified before fulfillment starts
  readonly fulfillmentCutoff: Date;
}
```

## Reverse-Engineering as a Practice

Instead of fighting this reality, embrace it. Reverse-engineering DDD means systematically extracting domain knowledge from existing code:

- **Map the implicit boundaries.** Look at which modules change together. Those are your real bounded contexts.
- **Find the hidden aggregates.** Transaction boundaries reveal what the system considers a consistency unit.
- **Extract the ubiquitous language.** The names in your code ARE your domain language, whether you like them or not.

## Making It Systematic

The key is to treat reverse-engineering as a repeatable process, not a one-time archaeology expedition. Every sprint, ask: what did this code teach us about our domain that we didn't know before?

Build feedback loops between the code and your domain model. When you discover an implicit rule in the code, make it explicit. When you find a misleading name, rename it and update the team's vocabulary.

## Conclusion

The best domain models aren't designed top-down — they're discovered through iterating between code and understanding. Stop treating your codebase as an implementation detail and start treating it as a domain knowledge repository.
