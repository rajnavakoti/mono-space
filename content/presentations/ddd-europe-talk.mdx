The builders were gone. The documentation was gone. But **the machine remembered everything.**

<Callout>
Legacy systems are the same. They look like technical debt. They're actually dense archives of domain knowledge — written under deadline pressure, tested by millions of real transactions, validated over decades.

**The architects left. The documentation went stale. But the system kept running. And it remembers everything.**
</Callout>

<Notes>

But inside, encoded in the schemas, the contracts, the error codes, the git history — is everything the original builders knew about the domain. This talk is about how to read what the system has been trying to tell you."
</Notes>

## What is "Legacy"?

<Callout>
Legacy isn't about age.

**It's about lost knowledge.**

A system is legacy when the people who understood it are no longer available to explain it.
</Callout>

<Notes>
"Before we go further — let me define legacy. I don't mean old code. A 6-year-old microservice can be legacy if the team that built it has turned over. A 30-year-old monolith might not be legacy if the original team still maintains it. Legacy is about KNOWLEDGE DECAY — when the understanding of why the system works the way it does has left the building."
</Notes>

## The question that started this talk

<Callout>
What if you could **extract a draft domain model** from the artifacts the system already produces — **before** you run your first Event Storming session?

Not to replace the workshop. To **give it a head start.**

To ask *"is this how it works?"* instead of *"how does this work?"*
</Callout>

<Notes>

Not replace Event Storming. Give it a head start. Arrive with evidence instead of blank sticky notes. Ask 'is this how it works?' instead of 'how does this work?' — because people are MUCH faster at reacting to evidence than creating from scratch."
</Notes>

## Dead Boundary vs Boundary Violation

> *Not all coupling is the same*

<Split ratio="2:1">
<div>

<CodeBlock title="shipment-carrier · coupling evidence" code="# SHIPMENT SERVICE defines:&#10;  ShipmentInfo:&#10;    carrierId: string&#10;    carrierName: string&#10;    trackingNumber: string&#10;    estimatedDelivery: date&#10;&#10;# CARRIER SERVICE defines:&#10;  Shipment:&#10;    shipmentId: string&#10;    orderId: string        # from Shipment&#10;    recipientId: string    # from Consignee&#10;    carrierName: string&#10;    trackingNumber: string&#10;&#10;# Same concept. Two owners.&#10;# Circular references." />

</div>
<div>

<Callout type="danger" label="DEAD BOUNDARY">
Both services define **Shipment** schema. Both reference each other. **Circular.**

**Duplicate ownership** of the same concept.

→ **MERGE** into one context or build a proper **ACL**.
</Callout>

</div>
</Split>

<Split ratio="2:1">
<div>

<CodeBlock title="shipment-inventory · coupling evidence" code="# SHIPMENT SERVICE references:&#10;  Order:&#10;    warehouseId: string    # from Inventory&#10;&#10;# INVENTORY SERVICE references:&#10;  Reservation:&#10;    orderId: string        # from Shipment&#10;&#10;# Different concepts. Shared IDs only.&#10;# No schema duplication." />

</div>
<div>

<Callout type="warning" label="BOUNDARY VIOLATION">
Share IDs only. **No duplicate schemas.**

**Order ≠ Reservation.** Different concepts.

The **integration pattern is wrong**, not the boundary.

→ **FIX the pattern** (replace co-write with saga/events)
</Callout>

</div>
</Split>

<Notes>
"Not all coupling is the same. Look at the evidence side by side. Shipment and Carrier both define a Shipment schema — carrierId, trackingNumber, same fields, two owners. Circular references. That's a dead boundary — same concept, two services pretending to be separate. Either merge or build an ACL. Now Shipment and Inventory — they share IDs only. orderId and warehouseId. But Order is not the same concept as Reservation. These are genuinely different domains coupled by the wrong integration pattern. Don't merge — fix the pattern. Replace the shared transaction with events or a saga."
</Notes>
