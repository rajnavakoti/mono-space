## DDC — Agent Onboarding

<Split ratio="1:1">
<div>

When a new team member joins, you don't hand them a 500-page doc and say "learn everything."

You give them a **problem**. They try. They fail. They ask questions. You answer. They try again. They succeed.

**Each problem reveals what they need to learn.**

DDC does the same for AI agents:

1. **Give** the agent a real problem
2. **Watch** it fail — it generates a demand checklist
3. **Fill** only the gaps it identified
4. **Curate** into structured, reusable entities
5. **Re-run** — agent succeeds with domain-specific reasoning
6. **Repeat** — knowledge compounds across problems

</div>
<div>

*(Right side: Lifecycle diagram — circular, not linear:*
*Six nodes in a circle connected by arrows:*
*PROBLEM → AGENT FAILS → DEMAND CHECKLIST → HUMAN FILLS GAPS → CURATE ENTITIES → AGENT SUCCEEDS*
*Center text: "30-60 min per cycle"*
*An outer arrow from AGENT SUCCEEDS back to PROBLEM labeled "Next problem"*
*Style: heavy borders, each node is a bordered box, arrows are thick.)*

</div>
</Split>

<Notes>
"Think about how you onboard a new team member. You don't hand them a 500-page document. You give them a problem. They try. They fail. They ask questions. You answer. They learn exactly what they needed — nothing more, nothing less. DDC does the same thing for agents. Give it a real problem. It fails. But it doesn't just fail — it tells you exactly what knowledge it's missing. A domain expert fills those gaps. The knowledge gets curated into structured entities. The agent tries again and succeeds. Each cycle takes 30 to 60 minutes. And the knowledge compounds."
</Notes>

## The TDD Parallel

If you've practiced Test-Driven Development, you already know this:

| TDD | DDC |
|-----|-----|
| Write a failing test | Give agent a failing problem |
| Write minimum code to pass | Curate minimum context to succeed |
| Test passes | Agent produces correct output |
| Refactor | Validate and tighten entity definitions |
| Next test | Next problem |

**You don't write all the tests upfront.**
**You don't curate all the context upfront.**

You let failures tell you exactly what's missing.

<Notes>
"Who here practices TDD? Good. You already understand DDC. In TDD, a failing test tells you what code to write. In DDC, a failing agent tells you what knowledge to curate. Same principle. Different domain. You don't write all the tests upfront — and you don't curate all the context upfront. Failures drive both."
</Notes>

## Where It Breaks

Honest section. No sugar coating.

- **We're early.** This is a direction, not a destination.
