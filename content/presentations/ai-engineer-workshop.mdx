## The TDD Parallel

If you've practiced Test-Driven Development, you already know this:

| TDD | DDC |
|-----|-----|
| Write a failing test | Give agent a failing problem |
| Write minimum code to pass | Curate minimum context to succeed |
| Test passes | Agent produces correct output |
| Refactor | Validate and tighten entities |
| Next test | Next problem |

**You don't write all the tests upfront. You don't curate all the context upfront.**

<Notes>Has anyone here practiced TDD? Good. Then you already understand DDC. In TDD, you write a failing test, then the minimum code to pass. You don't write all the tests upfront. DDC is the same principle applied to knowledge. Give the agent a problem. It fails. Its failure tells you EXACTLY what knowledge to curate. Curate the minimum. Agent succeeds. Next problem. You don't curate all the context upfront. You let failures tell you what's missing.</Notes>

---

## One DDC Cycle

<Split ratio="1:1">
<div>

**1. Give agent a real problem**
An incident, a design question, a ticket.

**2. Agent fails (RED phase)**
Produces a Demand Checklist — exactly what's missing.

**3. Human fills the gaps (GREEN phase)**
Domain expert provides the missing knowledge.

**4. Knowledge gets curated**
Structured entities, connected, placed in the repo.

**5. Agent re-answers**
Specific, accurate, domain-aware.

</div>
<div>

**6. Human reviews**
Expert validates. Corrections go back into entities.

**The cycle takes 30-60 minutes.**

**After 20-30 cycles, the agent knows your domain.**

Each new problem needs less new context because previous cycles already documented overlapping knowledge.

</div>
</Split>

<Notes>Here's what one cycle looks like. You give the agent a real problem — an incident, a Jira ticket, an architecture question. The agent tries with zero context and fails. But it doesn't just fail — it produces a Demand Checklist. A structured list of exactly what knowledge is missing. A domain expert fills those gaps. The knowledge gets curated into structured entities. The agent re-answers — and this time it's specific, accurate, domain-aware. The whole thing takes 30 to 60 minutes. And here's the key: after 20-30 of these cycles, the agent knows your domain. Because each new problem overlaps with previous ones.</Notes>

## The Context Lake

Your curated knowledge lives in a **Git repo**.

<Split ratio="1:1">
<div>

### Why Git?
- Version controlled — every change tracked
- PRs for human review — no unchecked additions
- Branching — experiment without breaking production
- Your engineers already know this workflow
- Machine-readable AND human-readable

</div>
<div>

### Structure
```
domain-knowledge/
  entities/
    systems/
    processes/
    jargon-business/
    jargon-tech/
    data-models/
  diagrams/
  decisions/
```

</div>
</Split>

<Notes>The curated knowledge lives in a Git repo. Why Git? Because your engineers already know the workflow. Version control, pull requests, branching, code review — it's the same process they use for code, applied to knowledge. Every entity is a markdown file with YAML frontmatter for relationships. Human-readable, machine-parseable. No special database, no vendor lock-in.</Notes>

---

## Entity Format

<CodeBlock title="system-entity.md" highlight="1-7" code={'---\ntype: system\nid: order-management\nname: Order Management Service\nstatus: active\nrelated_systems: [fleet-management, payment-gateway]\nowned_by: checkout-team\n---\n\n# Order Management Service\n\n## Overview\nHandles order lifecycle from placement to fulfillment.\nReceives orders from web/mobile/POS channels.\n\n## Key Behaviors\n- Validates inventory before confirmation\n- Splits orders across warehouses when needed\n- Sends DELIVERY_READY event to Fleet Management'} />

<Notes>Each entity looks like this. YAML frontmatter at the top — type, ID, status, relationships to other entities. Then markdown prose — overview, key behaviors, anything the agent needs to know. The relationships in the frontmatter are what make this a knowledge BASE, not just a knowledge DUMP. The agent can traverse from one entity to related entities. "I'm looking at Order Management — what systems does it connect to? Let me read Fleet Management next."</Notes>

## The Meta Model — Your Store Map

Think of it like the map at the entrance of an IKEA store.

LEO needs to know: **where things are, how they connect, and how to get from one place to another.**

<Split ratio="1:1">
<div>

### 13 Entity Types
- Systems, APIs, Platforms
- Processes, Business Events
- Data Models, Data Products
- Teams, Personas, Capabilities
- Business Jargon, Tech Jargon
- Offerings

</div>
<div>

### Relationships
- `related_systems: [a, b]`
- `implements_capability: x`
- `depends_on: [y, z]`
- `owned_by: team-id`

Agents follow relationships to build context for any problem.

</div>
</Split>

<Notes>Think of the meta model like the store map at the entrance of an IKEA. You know that map? It shows you where everything is and how departments connect. LEO needs the same thing for your domain. The meta model defines 13 entity types and the relationships between them. It's not mandatory — you can start simple with just systems and jargon. But as the knowledge base grows, the relationships become powerful. An agent working on an incident can trace from a system to its dependencies to the team that owns it to the business process it supports.</Notes>

I'm going to be honest about what doesn't work.
